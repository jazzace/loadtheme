URLTextSearcher Cheat Sheet

result_output_var_name:
    defaults to "match", but can be used to assign a variable name used by another processor(s)
re_pattern:
    Regular expression (Python) to match against page;
    (‚Ä¶) limits what is assigned to the output variable;
    (?P<name>‚Ä¶) assigns the found sub-pattern to a variable named "name"
        [Plist recipes: (?P&lt;name&gt;‚Ä¶)]
url:
    URL to download OR path to file (using file:///‚Ä¶)

result_output_var_name

I‚Äôll start with a tip that is about how you can better leverage a documented feature. The superpower of the URLTextSearcher processor (and my derivative of it for use on a single text string, TextSearcher) is the ability to use a regular expression to find the text you need. This is commonly used to extract a download URL from a web page, but you can actually extract any text that a regular expression can find for you. My favourite feature of this processor is that you can specify the name of the variable you want AutoPkg to assign to the result of the regular expression search using the result_output_var_name argument. For example, if you set result_output_var_name to a value of url and the next processor is URLDownloader, you do not have to specify the url argument, since it already has a value for url (assuming, of course, that you grabbed a full URL with your search pattern).

Selecting Sub-patterns

The URLTextSearcher processor only finds the first match for the regular expression, which is often very useful (e.g., a web page listing all the releases of the app starting with the most recent) but can also be challenging (e.g., when the first download link on the page is not the URL you are looking for). For those times when you need to provide context within your regular expression in order to grab the correct string, parentheses (i.e., the things surrounding this text) can tell AutoPkg, ‚ÄúI only want you to return this part.‚Äù Here is an example from my repo of this technique:

<key>Processor</key>
<string>URLTextSearcher</string>
<key>Arguments</key>
<dict>
	<key>re_pattern</key>
	<string>/AUTOMATIC1111/stable-diffusion-webui/releases/tag/v([\d.]+)"</string>
	<key>result_output_var_name</key>
	<string>version</string>
	<key>url</key>
	<string>https://github.com/AUTOMATIC1111/stable-diffusion-webui/releases</string>
</dict>

In this case, I am trying to grab the version number of the latest release from the GitHub Releases page for this project.[1] But if I just search for the first thing that looks like a version number (e.g., [\d.]+, which will locate the first combination of digits and dots it sees), I am unlikely to find what I need. So I look at the HTML in the area of the page where the version number is specified and I include enough of the surrounding HTML in my search pattern to make certain that the desired version is contained within.[2] I then add parentheses around the part of the expression that captures the version number to signal that I want to capture only that sub-pattern: ([\d.]+)

URLTextSearcher recognizes that I only want the sub-pattern and returns that value. In addition, by setting the result_output_var_name argument to version, any subsequent processor that accepts an argument for version will use that value.

Named Sub-patterns

Sometimes, however, there are times when you want to capture more than one value at the same time when using URLTextSearcher. People used to working with regular expressions that use one or more sets of parentheses to identify sub-patterns might be tempted to reference them using \1, \2, etc., but that doesn‚Äôt work in this context. What does work is giving each pattern a name. The earliest recipe writers (whose recipes I copied to learn about this technique) knew that they could name a sub-pattern because it was supported in Python. The result of a named sub-pattern becomes a variable available to AutoPkg. As you can imagine, this can be very powerful.

The syntax is to add ?P<name> just inside the left parenthesis, where name is replaced by a meaningful variable name.[3] Here is an example of capturing a single sub-pattern using this technique from a recipe that downloads an Apple Support Article:

<key>Processor</key>
<string>URLTextSearcher</string>
<key>Arguments</key>
<dict>
	<key>re_pattern</key>
	<string>meta property="og:title" content="(?P&lt;arttitle&gt;.*)"</string>
	<key>url</key>
	<string>file://%RECIPE_CACHE_DIR%/downloads/%ARTICLE%</string>
</dict>

Note that I had to change the angle brackets (< >) to their XML entities (&lt; &gt;) because the recipe is in plist form. It is a little easier to read in YAML:

- Processor: URLTextSearcher
  Arguments:
    re_pattern: meta property="og:title" content="(?P<arttitle>.*)"
    url: file://%RECIPE_CACHE_DIR%/downloads/%ARTICLE%

In this case, my regular expression looks in the header of the HTML page and grabs the ‚Äúcontent‚Äù from between the double-quotes and uses that as the value of the variable arttitle. Of course, from what we learned in the previous part of this article, I could have also obtained the same result by not using the ?P syntax and instead just using parentheses and assigning result_output_var_name a value of arttitle: 

  Arguments:
    re_pattern: meta property="og:title" content="(.*)"
    url: file://%RECIPE_CACHE_DIR%/downloads/%ARTICLE%
    result_output_var_name: arttitle

Because I haven‚Äôt always had the understanding of sub-patterns that I do now, you will see both forms in my recipes when I need to use a sub-pattern. Nevertheless, each form correctly assigns the desired value to arttitle.

Capturing More Than One Sub-pattern

This technique of named sub-patterns is how you are able to capture more than one value at a time. In the the same recipe I just cited, there is an example of gathering multiple sub-patterns from a single processor call. I will show it here converted to YAML:[4]

- Processor: URLTextSearcher
  Arguments:
    re_pattern: 'publish_date: "(?P<MM>\d\d)(?P<DD>\d\d)(?P<YYYY>\d{4})"'
    url: file://%RECIPE_CACHE_DIR%/downloads/%ARTICLE%

In this case, the web page stores the date in MMDDYYYY format. For my purposes, I want the date in ISO 8601 format, specifically YYYY-MM-DD. So I use named sub-patterns to grab the 2-digit month (which I name MM), 2-digit day of the month (DD), and the 4-digit year (YYYY). When I run this recipe on a page with a publication date of 08192024 (i.e., 2024 August 19), URLTextSearcher returns:[5]

{'Output': {'DD': '19', 'MM': '08', 'YYYY': '2024', 'match': '2024'}}

I then was able to use these new values in the filename of what I wanted for output:

- Processor: Copier
  Arguments:
    destination_path: '%SAVE_LOCATION%/%YYYY%-%MM%-%DD% %ARTICLE% %arttitle%.html'
    overwrite: true
    source_path: '%pathname%'

In this case, %ARTICLE% is an input variable specifying the number of the Apple Support Article we downloaded. All the rest of the variables being substituted were obtained from the regular expressions as arguments to URLTextSearcher, as shown above.

It Doesn‚Äôt Have To Be a Web Page

The observant reader will notice that the url argument for a number of the cited examples starts with file:// rather than https://. URLTextSearcher can work on a local file (that it has permission to read) just as well as a web page it downloads from the Internet if you use the file:/// syntax.[6] This can be used to search a web page multiple times or extract data from a file that one of the other core processors can‚Äôt handle. Both recipes I cite in this article are real oddballs in that they do not deal with installers or apps ‚Äî StableDiffusionWebUI.download downloads Python and Javascript code and AppleSupportArticleDateTitle.download literally downloads a web page (to track how often it changes and to cache copies as they are posted). Yet this makes them perfect for these techniques.
